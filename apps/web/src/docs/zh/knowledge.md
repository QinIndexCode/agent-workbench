# 知识库

知识库存放可复用说明和导入资料，供之后检索使用。它是资料库的一部分，适合沉淀稳定背景材料，而不是替代当前任务的实时文件观察。

## 适合放什么

- 设计说明
- 架构事实
- 需要以后再搜索的导入文档
- 能帮助后续任务的稳定解释
- 外部资料摘要、供应商行为说明、项目运行手册

## 不要和实时源码混淆

`knowledge_search` 搜索的是资料库里保存的内容，不是当前工作区的实时文件状态。需要核对当前源码时，仍应使用实时文件工具。

## 入口

- Web：进入 **资料库 → 知识库**，可以新建条目、上传文本类文件、按标签和状态筛选、测试检索结果。
- 输入框：`/knowledge` 打开知识库；`/knowledge <问题>` 会提交一个要求优先使用 `knowledge_search` 的任务请求。
- CLI：`aw knowledge list|add|upload|search|reindex|update|delete|models|download-model` 使用同一套 server API，不直接写 SQLite。

## 项目作用域

知识条目带有 `projectId`。任务内的 `knowledge_search` 默认使用当前任务文件夹对应的项目作用域；CLI 可用 `--project <id>` 指定。跨项目复用资料时，先确认它是否真的是全局稳定事实。

## 索引状态

- `pending`：已保存但还未完成索引
- `indexed`：可参与检索
- `failed`：索引失败，需要查看内容或重建索引
- `metadata_only`：内容为空或不可索引，只保留标题、标签、来源等元数据

修改内容、上传文件或清空内容后，系统会重建或清理对应索引。批量重建适合导入后检查召回质量，而不是日常每次任务前都运行。

## 检索质量检查

先用页面里的 **检索测试** 或 CLI 的 `aw knowledge search "<query>" --json` 看实际命中项、片段、匹配字段、召回等级和置信度。若搜索不到，优先检查标题、标签、来源和正文是否包含稳定关键词，再考虑本地模型。

## 智能检索方式

检索层会先生成一个本地查询计划，再进入同一套索引和重排链路。默认 `auto` 模式会保留原始问题，同时加入规范化查询、代码标识符拆分、常见领域别名和 step-back 查询；例如中文的权限/审批问题可以召回英文 `approval`、`permission`、`grant` 资料，`load knowledge index` 也能匹配 `loadKnowledgeIndex`。

这些增强是确定性的本地逻辑，不会额外调用 LLM，因此不会降低提示缓存命中率。需要排查召回原因时，使用 `aw knowledge search "<query>" --diagnostics --json` 查看查询变体、命中查询、召回等级和置信度；需要更严格的关键词匹配时，可用 `--mode keyword`。

## 高级检索

**Advanced** 区域可以下载 fastText 向量和 tiny ONNX 重排器。

建议在以下条件满足后再启用：

1. 内容本身已经整理干净
2. 标签足够有用
3. 你确实遇到了检索质量问题

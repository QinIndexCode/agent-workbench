# 任务管理

任务是 Agent Workbench 的核心组织单元，每个任务对应一次独立的会话上下文。

## 创建任务

- 点击侧边栏的 **添加任务** 按钮创建新任务
- 新任务会出现在侧边栏的任务列表中
- 点击任务标题可切换当前会话

## 目标完成模式

当你希望 Agent 追求可验证的结果，而不是只给出简短回答时，可以输入 `/goal <需求>`。目标完成模式会运行更久，持续展示验收标准、工具进度、文件变更和验证状态，并在探索、实现、验证之间继续推进，直到任务完成，或因上限、权限、模型异常、无进展恢复或用户中断而暂停。

启动 `/goal` 一定会弹出确认框。请仔细阅读后果提醒：该模式可能消耗更多模型额度，反复读写文件、运行命令或访问网络。选择 **Full risk** 会全局允许 destructive 操作，例如删除、覆盖和终止进程，必须额外勾选确认后才能启动。

## 输入指令

在输入框第一行输入 `/` 会出现可用指令。指令只改变当前请求如何进入现有任务系统，不会绕过权限、模型配置、附件处理或 server API。

### `/goal <需求>`

创建一个目标完成模式任务。适合“修复并验证”“深入审查”“补齐功能”“长链路测试”等需要持续推进的工作。启动前必须确认权限范围。

### `/plan <需求>`

创建或追加一个“先规划再确认”的请求。Agent 会先给出可见计划、验收标准、风险和需要确认的问题，然后等待你确认再继续。它不会直接开启目标完成模式，也不会自动允许写文件、运行命令或联网。

### `/review <范围>`

以审查模式提交请求。Agent 应先列出可复现问题、风险、行为回归和缺失测试，再修复能够确认的问题。

### `/verify <目标>`

以证据优先方式验证当前状态。适合确认某个功能、修复、CLI 命令或文档是否真实可用，并明确剩余风险。

### `/debug <问题>`

以调试闭环处理问题：复现或定位症状、找到最小根因、修复、回归验证。

### `/research <问题>`

用于需要当前资料或外部事实的研究请求。时间敏感信息应联网核验；项目内事实应引用实际文件、接口或测试证据。

### `/doc <请求>`

创建或更新文档，并要求先核对当前代码和既有文档边界。不能把计划写成已实现事实，也不能夸大协议或工具能力。

### `/knowledge [问题]`

无参数时打开 **资料库 → 知识库**。带问题时提交一个知识库请求，要求在需要历史资料时使用 `knowledge_search`，并在涉及当前源码时继续用实时文件工具核验。

### `/memory [请求]`

无参数时打开 **资料库 → 记忆**。带请求时检查或整理持久记忆，只保存稳定、可复用、已确认的信息。

### `/skill [请求]`

无参数时打开 **资料库 → Skills**。带请求时检查 Skill 的重复、冲突、适用范围和是否值得创建或合并。

### `/cache [范围]`

检查 LLM 请求缓存命中率、provider 配置和请求组装路径。该指令要求保持任务质量，不应为了缓存命中而删除必要工具、上下文或验证步骤。

### 导航指令

- `/docs` 打开文档中心
- `/settings` 打开设置中心
- `/model` 打开模型配置
- `/permissions` 打开权限审批
- `/search` 打开 Web 搜索 provider 配置
- `/mcp` 打开 MCP 配置
- `/integrations` 打开外部集成
- `/schedule` 打开定时任务
- `/prefs` 打开偏好设置
- `/curator` 打开 Curator

### `/help`

打开任务管理文档，查看当前支持的输入指令和边界。

### `//`

如果你只是想发送以 `/` 开头的普通文本，用 `//` 开头。系统会把它作为普通消息发送，而不是当成指令解析。

## 任务历史

- 所有任务会话会自动保存
- 侧边栏的任务列表支持滚动浏览
- 点击任务可查看完整的历史消息

## 任务执行流程

1. 用户发送消息
2. 系统组装上下文（历史消息 + 当前输入 + 附件内容）
3. 模型生成响应
4. 如响应包含工具调用，系统按权限策略处理
5. 工具执行结果返回模型，继续生成
6. 最终结果展示给用户

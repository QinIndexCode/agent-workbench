## 写在前面：Honest Engineering 的设计哲学

> "在 LLM 领域取得成功，不是要构建最复杂的系统，而是要构建适合您需求的系统。" —— Anthropic

### 我们的设计态度

SCC-Batch Engine 的设计遵循一个核心原则：**只有在简单方案不足时，才增加复杂性**。

这不是技术炫技的产物，而是对现有多智能体框架痛点的诚实回应。我们不追求"完美架构"，而是追求"能解决当下问题、且能适应未来变化"的架构。

**三个核心态度**：

1. **框架是拐杖，不是双腿** —— 理解底层机制，比使用任何框架都重要
2. **承认边界，而非强行适配** —— 一个好的架构，是知道自己"不做什么"的架构
3. **从简单开始，按需增加复杂度** —— 最成功的实现通常只是 LLM 在循环中基于环境反馈使用工具

### 当前实现状态边界（2026-05）

本文是研究与架构愿景文档，不是逐项已交付清单。后续阅读按以下标签解释：

- **Implemented**：代码中已有可运行实现。

当前 SCC 的主线强项是本机 Agent Workbench、风险审批、工具证据回灌、上下文组装、File State Table、experience/skill 学习雏形和 OpenAI-compatible provider 接入。批处理 DAG、完整 ContractParser/ContractValidator/SnapshotManager、MCP connector、多消息平台网关、完整远程执行后端、Python SDK、OpenTelemetry 完整兼容和生态迁移工具仍属于 planned 或 research，不应被描述为已完成。

**重要边界**：本文包含愿景和研究假设。产品验收以源码、测试、真实任务行为和 `architecture.md` 的当前边界为准。本文中涉及“检查点”“校验”“验证”的内容仅指结构契约、安全权限、工具 I/O 和工程测试，不允许恢复脚本式任务质量门禁。

### 2025-2026 年的行业启示

Anthropic 在《Building Effective Agents》2025 更新版中反复强调：

> "框架简化了入门，但可能创建额外的抽象层，掩盖底层细节，使调试更困难。"

LangChain 创始人更直接警告：2026 年应"放弃 workflow 编排工具"，转向面向未来的智能体框架。

**SCC-Batch 的回应**：我们不是在构建另一个编排框架，而是在探索一种**让模型原生能力最大化**的执行范式——用契约约束边界，用批处理释放效率，用最小上下文保证精准。

## 一、架构核心定位与设计初衷

### 1. 一句话定义

SCC-Batch Engine 全称 **Semantic Contract-based Batch DAG Engine**（语义契约式批处理DAG引擎），是一款**极简、高效、强可控的多单元协同执行引擎**。

它的核心突破是：尽量打破 LangGraph/CrewAI/AutoGen 等框架「单Agent = 单轮API调用」的先天瓶颈，通过**强语义契约约束**，让大模型在**单次/极少次API前向传播内，完成可验证 DAG 的多单元调度、执行、隔离与校验**。这是目标导向的执行范式；复杂工具任务由 runtime observation、permission approval、tool evidence 和 agent 自主纠错闭环支撑。

### 2. 解决的行业核心痛点

| 现有框架的核心痛点 | SCC-Batch 的解决思路 | 行业标杆的对应实践 |
|:---|:---|:---|
| **多轮调用开销爆炸**：复杂DAG需要N次API调用，延迟高、成本高 | 把多轮串行调用，压缩到单次/极少次调用内完成 | Claude Code「批量工具预规划+原子化执行」 |
| **上下文污染与串戏**：多Agent共享全局上下文，极易越权访问、信息泄露 | 用契约+权限分级，严格限制每个执行单元的上下文访问边界 | Cursor「精准可控的上下文锚定机制」 |
| **调度与推理割裂**：工程代码做调度，模型只做单步生成 | 把DAG调度规则固化到语义契约里，调度与推理一体化 | Claude Code「极简主循环+可控的Agent自主权」 |
| **落地门槛高**：需要复杂的工程配置、依赖安装 | 完全基于标准`chat/completions`接口，无需微调、无需额外权限 | Anthropic "从 LLM API 直接调用开始" |

### 3. 核心设计前提

整个架构建立在一个经过验证的核心假设之上：

> **通过标准化的强语义契约+不可绕过的规则约束，当前主流模型在约束良好的任务中可以显著提高契约遵守率，并在失败时提供可诊断、可修正的运行真相。**

这不是无条件保证，而是需要通过真实任务、benchmark、tool evidence、结构契约校验和用户可见结果持续验证的工程假设。关键在于：**给模型清晰的边界，并让运行时严格检查边界，它才有稳定改进的空间**。

**这与 Anthropic 的 ACI（Agent-Computer Interface）原则完全一致**：
- 工具定义要清晰，包含使用示例和边界情况
- 参数命名要直观，像给初级开发者写文档
- 防错设计（Poka-yoke），让错误难以发生

## 二、架构三大核心设计理念

### 理念1：契约优先，无契约不执行

所有执行逻辑、输入输出、权限边界、退出规则，全部用标准化的语义契约提前定义。模型只能在契约约定的范围内执行，没有任何模糊空间。

**行业标杆对齐**：

| 标杆 | 设计理念 | SCC-Batch 实现 |
|:---|:---|:---|
| Claude Code | 渐进式披露的分层上下文管理 | `INPUT_CONTRACT` 限制上下文访问边界，仅给Agent当前步骤最小必要的信息 |
| Cursor | 精准可控的上下文锚定机制 | 把上下文的控制权交给使用者，实现"需要什么，才能用什么" |
| Windsurf | 上下文隔离与作用域精准控制 | 通过明确的契约定义每个执行单元的作用域、可访问范围、可修改权限 |

**设计收益**：
- 在适配场景下降低大量无效 token 占用，具体比例以 benchmark 报告为准
- 大幅提升核心规则的遵守率
- 从根源上杜绝上下文污染、越权访问

### 理念2：批处理优先，极致压缩调用次数

核心目标是「用最少的API调用完成最多的任务」。能在单次调用里完成的，绝对不拆成多轮。

**行业标杆对齐**：

| 标杆 | 设计理念 | SCC-Batch 实现 |
|:---|:---|:---|
| Claude Code | 批量工具预规划+原子化执行 | 第1次API输出工具调用清单，批量执行后再第2次API完成剩余单元 |
| GitHub Copilot Workspace | 任务级拆解与可干预规划 | DAG依赖固化到Prompt，单次调用完成调度 |

**设计收益**：
- API调用次数从 N 次尽量压缩到少数关键回合
- 端到端延迟按任务类型和工具需求验证，不做固定倍数承诺
- 彻底解决"工具调用与批处理的逻辑矛盾"

### 理念3：最小上下文与强隔离原则

每个执行单元只能拿到「完成自身任务必须的最小上下文」，严格限制访问范围。

**行业标杆对齐**：

| 标杆 | 设计理念 | SCC-Batch 实现 |
|:---|:---|:---|
| Cursor | 项目级语义索引与缓存机制 | 提前对可复用的资源做结构化、索引化处理，用"按需查询"替代"全量灌入" |
| Trae | 上下文权限边界可视化 | 清晰展示AI当前可访问的上下文范围、已经引用的内容、可操作的权限 |

**设计收益**：
- 兼顾上下文的完整性和轻量化
- 从根源上杜绝上下文污染、跨单元串扰
- 降低 token 消耗，提升模型注意力和规则遵守率

## 三、全架构分层与核心模块详解

```
┌─────────────────────────────────────────────────────────┐
│  1. 对外接口层（CLI / Python SDK / HTTP API）           │
├─────────────────────────────────────────────────────────┤
│  2. 契约与DAG定义层（核心语义模型、规则定义）           │
├─────────────────────────────────────────────────────────┤
│  3. 核心引擎层（Prompt构建、调度、解析核心逻辑）        │
├─────────────────────────────────────────────────────────┤
│  4. 校验与容错层（契约校验、错误处理、快照管理）        │
├─────────────────────────────────────────────────────────┤
│  5. 工具与生态兼容层（工具调用、MCP/生态适配）          │
├─────────────────────────────────────────────────────────┤
│  6. 可观测性与调试层（执行追踪、可视化、问题归因）      │
└─────────────────────────────────────────────────────────┘
```

### 第1层：对外接口层

核心职责：**降低使用门槛，提供多场景的接入方式**。

| 接入方式 | 适用场景 | 设计原则 |
|:---|:---|:---|
| **Python SDK** | 代码化定义，快速迭代 | 装饰器/类定义，一行代码触发执行 |
| **CLI 命令行** | 脚本化、自动化场景 | YAML/JSON配置，终端直接执行 |
| **HTTP API** | 企业级集成 | 服务化封装，供其他系统调用 |
| **配置管理** | 环境切换 | 支持本地Ollama与云端API无缝切换 |

**行业标杆对齐**：

| 标杆 | 设计理念 | SCC-Batch 实现 |
|:---|:---|:---|
| JetBrains AI Assistant | 全流程场景的无缝嵌入 | 把AI能力嵌入到开发者的每一个操作环节，无需打开独立的聊天窗口 |
| GitHub Copilot Workspace | 原生协作流程无缝集成 | 直接对接GitHub的Issue、PR、代码仓库，完全融入开发者现有的协作流程 |

**设计启示**：好的能力设计，是"让用户在需要的时候，能最便捷的用到"，而非做一个独立的、割裂的功能模块。

### 第2层：契约与DAG定义层（核心创新）

这是整个架构最核心的创新部分，所有的执行、调度、隔离、校验，都基于这一层的定义。

#### 核心1：Agent Unit 语义契约模型（5大强制字段）

每个Unit对应DAG里的一个节点，所有行为完全由契约约束。

| 强制字段 | 核心作用 | 设计目的 | 行业标杆对齐 |
|:---|:---|:---|:---|
| `UNIT_ID` | 单元唯一标识 | 定位锚点，用于DAG依赖绑定、结果解析 | Claude Code「元数据先行」 |
| `TASK_SCOPE` | 任务边界、核心职责 | 明确"这个单元要做什么，不做什么" | Anthropic ACI「工具定义要清晰」 |
| `INPUT_CONTRACT` | 输入约定、可访问范围 | **解决上下文污染、越权的核心** | Claude Code「渐进式披露」+ Cursor「上下文锚定」 |
| `OUTPUT_CONTRACT` | 输出约定、JSON Schema | 规范输出格式，为校验提供标准 | GitHub Copilot Workspace「验证-修复闭环」 |
| `EXIT_CONDITION` | 完成标准、退出条件 | 避免执行不完整、提前终止 | Claude Code「闭环式自我校验」 |

#### 核心2：三级权限控制模型

| 权限级别 | 访问范围 | 适用场景 | 行业标杆对齐 |
|:---|:---|:---|:---|
| `GLOBAL` | 全局所有Unit可访问 | 全局目标、基础配置、公共参数 | Claude Code「分层级的安全权限护栏」 |
| `DEPENDENCY` | 只有直接依赖的下游Unit可访问 | 上游核心结果，避免无关单元读取 | Windsurf「上下文隔离与作用域精准控制」 |
| `PRIVATE` | 只有当前Unit自己能访问 | 中间变量、调试信息、不传递内容 | Cursor「最小范围增量修改范式」 |

#### 核心3：DAG 依赖定义模型

支持3种核心流程，与 Anthropic 五种工作流模式完美对应：

```
串行流程：A → B → C（需求分析 → 技术方案 → 代码生成）
├── 对应：提示词链（Prompt Chaining）
└── 特点：任务可清晰分解为固定子任务，牺牲速度换取更高准确性

并行流程：A、B、C 并行 → D 汇总（多维度数据分析 → 汇总报告）
├── 对应：并行化（Parallelization）
└── 特点：子任务相互独立，可并行执行，用空间换时间

条件分支：A → 判断 → B 或 C（复杂度高 → 详细设计 / 直接生成代码）
├── 对应：路由（Routing）
└── 特点：对输入分类，引导至专门的处理流程，关注点分离
```

**整体架构对应**：协调器-工作者（Orchestrator-Workers）的批处理优化版
- 中央 LLM 动态分解任务，分配给工作者 LLM，综合结果
- 子任务不是预定义的，而是由协调器根据具体输入动态确定

### 第3层：核心引擎层（架构的"大脑"）

#### 模块1：PromptBuilder 契约提示词构建器

**整个引擎最核心的模块之一**，直接决定模型能不能严格遵守契约。

核心设计细节：

| 设计点 | 具体实现 | 行业标杆对齐 |
|:---|:---|:---|
| 契约规则最高优先级 | 用"不可绕过的强制规则"开头，避免模型在长上下文里忽略约束 | Claude Code「安全规则必须放在系统提示的最高优先级」 |
| 正例+反例 | 告诉模型正确的输出格式、错误的格式是什么 | Anthropic ACI「包含使用示例和边界情况」 |
| DAG依赖顺序排列 | 引导模型按顺序执行 | Claude Code「极简主循环」——把任务拆解、步骤调度的自主权交给模型 |
| 渐进式披露 | 只包含和本次执行相关的契约内容 | Claude Code「元数据先行→按需加载执行资源」的三级加载机制 |

#### 模块2：Model Invoker 模型调用器

核心职责：标准化对接各类大模型接口，完全基于标准`chat/completions`格式。

**设计原则**：上层逻辑无需修改，一键切换模型。这是"框架是拐杖，不是双腿"的体现——理解底层API，而非依赖框架抽象。

**行业标杆对齐**：
- JetBrains AI Assistant「隐私优先的部署与合规设计」——支持本地模型部署、企业级私有部署
- Anthropic 建议「从 LLM API 直接调用开始，理解底层原理后再使用框架」

#### 模块3：ContractParser 契约结果解析器

核心职责：把模型生成的大段文本，按`UNIT_ID`拆分成每个单元的独立输出。

核心设计：
- 基于格式标记精准拆分，不会出现内容错配
- 自动提取输出字段，和`OUTPUT_CONTRACT`做匹配
- 支持增量解析，哪怕生成中断也能提取已完成结果

**行业标杆对齐**：Cursor「最小范围增量修改范式」——执行单元的粒度越细，容错性越强，出错后的修复成本越低。

#### 模块4：DAG Scheduler 调度器

核心职责：处理DAG的依赖关系、执行顺序管控、分支逻辑判断。

| 场景 | 调度策略 | 行业标杆对齐 |
|:---|:---|:---|
| 纯批处理 | DAG依赖固化到Prompt，单次调用完成调度 | Claude Code「极简主循环」——放弃硬编码DAG、固定分支流程 |
| 带工具调用 | 拆分执行阶段（规划→工具批量执行→结果批处理） | Claude Code「批量工具预规划+原子化执行」 |
| 错误重试 | 调度失败单元的增量重试，无需重跑已成功单元 | Windsurf「长任务的断点续跑与状态管理」 |

### 第4层：校验与容错层（生产级保障）

#### 核心模块1：Contract Validator 契约校验器

校验维度全覆盖：

| 校验维度 | 检查内容 | 行业标杆对齐 |
|:---|:---|:---|
| **格式校验** | 是否符合JSON Schema，必填字段是否缺失 | GitHub Copilot Workspace「内置的验证-修复闭环」 |
| **顺序校验** | 是否按DAG依赖顺序执行，有无乱序 | Claude Code「闭环式自我校验与迭代机制」 |
| **权限校验** | 是否读取了`INPUT_CONTRACT`约定之外的内容 | Windsurf「上下文隔离与作用域精准控制」 |
| **退出条件校验** | 输出是否满足`EXIT_CONDITION`定义的完成标准 | Claude Code「必须给Agent明确的、可执行的验证规则」 |

**对应 Anthropic 模式**：评估器-优化器（Evaluator-Optimizer）
- 一个 LLM 生成响应，另一个 LLM 提供评估和反馈，形成迭代循环
- 关键在于评估器的能力——它必须比生成器更"懂"什么是对的

#### 核心模块2：Snapshot Manager 快照与状态管理器

核心能力：

| 能力 | 说明 | 行业标杆对齐 |
|:---|:---|:---|
| 全局状态池 | 存储所有校验通过的Unit输出，严格按三级权限控制访问 | Windsurf「上下文隔离与作用域精准控制」 |
| 单元级快照 | 每个Unit校验通过后立即固化，后续出错可直接复用 | GitHub Copilot Workspace「可回溯的执行轨迹与变更对比」 |
| 断点续跑 | 执行中断后可基于最新快照继续执行 | Windsurf「长任务的断点续跑与状态管理」 |
| 执行轨迹记录 | 完整记录每个Unit的执行状态、输入输出、校验结果 | Cursor「思考过程全透明可追溯」 |

#### 核心模块3：Error Handler 分级错误处理

| 错误类型 | 处理策略 | 行业标杆对齐 |
|:---|:---|:---|

**行业共性设计哲学**：从"全流程重跑"到"最小范围增量修复"
- 针对长链路任务，做细粒度的单元拆分、状态快照、分级校验
- 彻底避免"一步错，全流程重跑"的低效问题，大幅提升容错性和执行效率

## 四、端到端完整执行流程

### 场景1：纯逻辑DAG，无工具调用（极致批处理）

**全程只需要1次API调用**，完整流程如下：

```
用户定义 → 前置校验 → Prompt构建 → 单次模型调用 → 结果解析 → 契约校验 → 结果输出
```

**关键设计点**：

| 阶段 | 核心动作 | 设计目的 |
|:---|:---|:---|
| 前置校验 | 检查契约字段完整性、DAG依赖合法性、Schema有效性 | 避免无效API调用，节省成本 |
| Prompt构建 | 把所有契约、规则、格式示例整合成强约束提示词 | 引导模型按DAG顺序执行，确保输出符合契约 |
| 单次调用 | 一次API调用完成所有单元执行 | 极致压缩调用次数，降低延迟和成本 |
| 解析校验 | 按UNIT_ID拆分输出，四维度校验 | 确保输出符合契约，为下游提供可靠输入 |

**关于并行DAG节点的执行语义**：在单次API调用的批处理模型下，并行节点并非真并行——Prompt将并行节点按 `UNIT_ID` 顺序排列，模型在单次推理中依次输出各节点的结果。这里的"并行"指**依赖关系上的并行**（节点间无数据依赖，可以任意顺序执行），而非计算资源上的并行。真正的计算并行（多个API调用同时发出）属于planned方向的优化，不在当前批处理范式的核心路径上。当前设计优先保证**单次API调用内完成全部调度**的核心优势。

**与 Anthropic 模式的对应**：提示词链（Prompt Chaining）的批处理优化版
- 将复杂任务分解为固定步骤，每步 LLM 调用处理前一步的输出
- 关键在于检查点的设计——如何在中间步骤验证流程是否正确

### 场景2：带工具调用的DAG（1+2次极限调用）

**全程仅2次API调用**，完美解决"工具调用需要中断，必须多轮调用"的矛盾：

```
定义与校验 → 规划与预执行(第1次API) → 批量工具执行 → 增量批处理(第2次API) → 校验输出
```

**关键设计点**：

| 阶段 | 核心动作 | 行业标杆对齐 |
|:---|:---|:---|
| 第1次API调用 | 完成不需要工具的单元 + 输出工具调用清单 | Claude Code「批量工具预规划」——先在推理阶段完成全链路规划 |
| 批量工具执行 | 外部执行器并行执行所有工具调用 | Claude Code「原子化执行」——外部执行器批量完成后，再把所有结果一次性喂给模型 |
| 第2次API调用 | 基于工具结果完成剩余单元 | Claude Code「极少次批量工具执行」——把API调用次数压到极限低的水平 |

**与 Anthropic 模式的对应**：协调器-工作者（Orchestrator-Workers）的批处理优化版
- 中央 LLM 动态分解任务，分配给工作者 LLM，综合结果
- 子任务不是预定义的，而是由协调器根据具体输入动态确定

## 六、架构的核心优势与适配场景

### 1. 不可替代的核心优势

### 3. 架构的边界（不适合的场景）

为了保持核心优势，架构天然不适合：

| 不适合场景 | 原因 | 替代方案 |
|:---|:---|:---|
| 实时人机交互、中途频繁人工干预的对话式Agent | 批处理范式与实时交互矛盾 | 传统对话式Agent框架 |
| 超长时间运行、需要持续感知环境变化的常驻Agent | 契约式执行与动态环境适配矛盾 | 自主Agent框架 |
| 每一步执行都需要人工确认、动态调整任务目标的强交互场景 | 批处理效率优势无法发挥 | 交互式工作流框架 |

**Honest Engineering 的态度**：承认边界，而非强行适配。一个好的架构，是知道自己"不做什么"的架构。

## 八、实践建议与落地路径

### 1. 从简单开始

```
第一步：单一 Prompt 评估
    ↓ 不满足需求
第二步：串行DAG（2-3个Unit）
    ↓ 不满足需求
第三步：并行/分支DAG
    ↓ 不满足需求
第四步：带工具调用的DAG
```

**Honest Engineering 原则**：只有在简单方案不足时，才增加复杂性。

### 2. 理解底层

如果使用 SCC-Batch，务必理解：

- 契约是如何被转换成Prompt的
- 模型是如何被引导按DAG顺序执行的
- 校验是如何确保输出符合契约的
- 错误是如何被分级处理的

**框架是拐杖，不是双腿。** 理解底层机制，才能在出问题时快速定位和修复。

### 3. 投资ACI（Agent-Computer Interface）

契约就是 SCC-Batch 的 ACI。好的契约设计：

- 字段命名直观，像给初级开发者写文档
- 包含使用示例和边界情况
- 防错设计（Poka-yoke），让错误难以发生

### 4. 设置检查点

在关键节点加入：

- 自动验证：Contract Validator 四维度校验
- 人工审核：高危操作申请用户确认
- 状态快照：每个Unit完成后固化状态

### 5. 持续迭代

基于评估结果不断优化：

- 契约定义是否清晰
- DAG依赖是否合理
- 权限控制是否精准
- 错误处理是否完善

### 6. 拥抱 MCP

使用标准化协议连接工具：

- 原生对接 MCP 生态
- 快速复用全生态工具集
- 避免重复造轮子

### 7. 善用 Skills

积累可复用的能力模块：

- 把高频场景的最佳实践沉淀为标准化模板
- Agent Unit 契约可封装为可复用 Skill
- 降低用户的使用门槛，保证执行效果的稳定性

## 十、总结：Honest Engineering 的最终回答

SCC-Batch Engine 不是要构建最复杂的系统，而是要构建**适合特定需求**的系统。

**我们的回答**：

1. **关于复杂性**：只有在简单方案不足时，才增加复杂性。用契约替代复杂的工程化流程，用批处理替代多轮调度。

2. **关于框架**：框架是拐杖，不是双腿。理解底层API，比使用任何框架都重要。SCC-Batch 基于 标准 chat/completions 接口，无额外依赖。

3. **关于未来**：2026 年是"长任务 Agent 元年"。SCC-Batch 的契约式批处理范式，正是面向这个未来的探索——让模型原生能力最大化，而非被工程化流程束缚。

> **最终目标**：在 LLM 领域取得成功，不是要构建最复杂的系统，而是要构建适合您需求的系统。技术是手段，不是目的。最好的架构，是能解决当下问题、且能适应未来变化的架构。
